## 一、核心架构（三种队列）
1. 业务队列：接收正常业务消息。
2. 重试队列：处理临时性错误，并为其配置死信队列。
3. 死信队列：所有重试失败后，消息的最终统一入口。

## 二、处理流程
* 业务队列（消费失败）  →  手动发送到延迟交换机（指数增长延迟时间）手动NACK 不重新入队列（requeue=false）  →  重试队列（重试失败计算recount超过则进入死信）Reject 不重新入队列  →  死信队列消费者拿到消费失败的记录，存入mq_fail 表中，记录业务id，业务类型，发送的消息体。  →  存入数据库  →  前端平台展示消息管理。
* confirmCallback 成功推送交换的回调，ack =false ，则存入mq confirm failure表中
